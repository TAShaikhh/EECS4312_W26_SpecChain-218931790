# Reflection

The manual pipeline produced the clearest personas because each persona was written from a small, reviewed set of related comments. That made the personas more specific and made it easier to write grounded requirements with low ambiguity. The automated pipeline was much faster and achieved the strongest review coverage and the highest raw traceability volume because it grouped the full cleaned dataset, but some outputs were too generic and occasionally drifted into unsupported claims or vague wording. The hybrid pipeline was the best balance for final quality because it started from the automated outputs and then corrected grouping mistakes, tightened persona grounding, and rewrote requirements and tests to be more defensible.

The most useful requirements came from the hybrid pipeline. The manual pipeline also produced strong requirements, but the hybrid pipeline benefited from seeing the automated themes first and then refining them into clearer, testable statements. The strongest traceability in raw quantity came from the automated pipeline because it covered the full cleaned dataset, while the clearest traceability in meaning came from the manual and hybrid pipelines because the links between review groups, personas, requirements, and tests were easier to justify by inspection.

The main problems in the automated outputs were vague personas, overgeneralized requirements, and occasional unsupported details introduced by the LLM. Some generated requirements were grammatically correct but still weak for validation because they were broad or not directly measurable. The hybrid step was useful because it kept the automation speed while removing those issues through manual review and correction.

Another important difference was how much human judgment each pipeline used. In the manual pipeline, most of the effort went into interpreting reviews carefully and deciding which feedback patterns were actually meaningful. That produced lower coverage, but it also reduced the chance of grouping unrelated reviews together. In the automated pipeline, clustering and prompting made it possible to process the entire cleaned dataset quickly, but the resulting groups sometimes mixed general praise with more specific complaints, which made the generated personas and requirements less precise. The hybrid pipeline improved this by keeping the automated starting point but applying human review where the grouping or wording was weak.

The validation tests also reflected these differences. Manual tests were the most detailed because they were written directly against intentionally crafted requirements. Automated tests were structurally consistent, but they depended heavily on the quality of the generated requirements, so vague requirements led to weaker scenarios. The hybrid tests were more useful than the automated ones because the revised requirements were narrower and easier to verify. Overall, this project showed that automation is valuable for scale and traceability, but human review is still important for grounding, requirement quality, and testability.
